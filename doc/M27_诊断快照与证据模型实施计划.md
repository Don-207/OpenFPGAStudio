# M27 诊断快照与证据模型实施计划

M27 是第六阶段 AI Debug 的数据基础里程碑。目标是把 Debug、Trace、Monitor、Profiler 和 Logic Analyzer 已有观测数据整理为稳定、可校验、可导出的 `Diagnostic Snapshot`，使后续规则和 AI 只依赖统一证据模型，不直接耦合各 Viewer 视图。

> 实施状态：已完成（2026-07-16）  
> 验收入口：`just m27-check`

## 0. 实施结果摘要

M27 已按“不重写既有 Viewer”的原则落地为独立 UMD 模块，并通过 `window.openfpgaViewerTest.diagnosticSnapshot` 最小接入当前 Viewer。模块可同时运行于浏览器和 Node.js，后续 M28 可直接消费其导出的 snapshot，而无需读取 Viewer 私有数组。

已实现的核心能力：

- 固化 `openfpga.diagnostic_snapshot` v1 顶层结构和 Evidence Item 必需字段。
- 从 Debug、Trace、Monitor、Profiler、Logic Analyzer 和 Viewer transport 内存状态只读提取证据。
- 生成基于 canonical 内容的稳定 `snapshot_id` 和 `evidence_id`。
- 保留 raw timestamp、统一 tick、wrap count 和 `exact/approximate/unknown` 对齐质量。
- 支持 session、时间窗口和指定 LA capture 的 snapshot 创建 API。
- 将 LA samples 压缩为关键边沿或稳定区间，并保留 capture/sample range 定位。
- 记录不完整 LA capture、missing range、错误和 transport/parser 计数。
- 提供 canonical JSON、SHA-256 生成/验证、重新导入和篡改检测。
- 提供 `local_export` 和 `provider_preview` 脱敏 profile；审计仅记录路径、动作和原因。
- 提供 evidence ID 到 `source_ref` 的定位 API。

## 1. 目标与边界

- 固化 `openfpga.diagnostic_snapshot` v1 schema 和 Evidence Item。
- 支持按当前会话、时间窗、Trace span、Profiler 异常或 LA capture 建立 scope。
- 统一时间基准，保留原始时间、换算时间及对齐质量。
- 为所有证据生成稳定 ID，并能由 `source_ref` 回到原始 Viewer 记录。
- 显式记录缺帧、缺 chunk、overflow、未知通道和近似时间对齐。
- 支持字段级脱敏、外发预览、canonical JSON 和 SHA-256 完整性摘要。
- 建立无硬件 fixture 和自动校验入口。

M27 不实现诊断规则、远程 Provider、AI 结论或完整 AI Debug 页面；仅提供后续里程碑可复用的数据层和最小测试钩子。

## 3. 数据契约

顶层必需字段：

```text
schema, schema_version, snapshot_id, created_at,
time_range, timebase, target, session_summary,
evidence, redaction, integrity
```

每条 evidence 必须包含：

```text
evidence_id, kind, source, timestamp, severity,
summary, data, source_ref
```

约束：

- `schema_version` 固定为 `1`，未知主版本必须拒绝。
- ID 在同一 canonical 输入和 scope 下可重复生成，不使用数组下标作为唯一来源。
- `source_ref` 至少包含视图、记录类型和稳定记录定位信息。
- 原始值保存在 `data`；派生摘要不能替代原始证据。
- 不可靠的时间关系标记为 `approximate` 或 `unknown`，不得强行排序为因果链。
- canonical hash 不包含自身 hash 字段，字段排序和数值序列化规则必须固定。

- [x] 定义 snapshot、evidence、timebase、redaction 和 integrity 结构。

- [x] 对未知可选字段保持前向兼容，对未知 schema 主版本明确报错。
- [x] 在 Python fixture 校验器中独立复算 canonical SHA-256，避免只依赖 JS 实现自证。

### M27.2 Snapshot Builder

- [x] 为 Debug、Trace、Monitor、Profiler、Logic Analyzer 及 Viewer transport 实现只读 adapter。
- [x] 实现时间窗口 scope、指定 LA capture scope、timestamp wrap 展开和稳定排序。
- [x] 生成 session summary、source counts、warning count 和完整性摘要。
- [x] LA 波形只提取关键边沿与稳定区间，同时保留 capture/sample range 定位。
- [x] 无法获得 timestamp 的记录保留为 `unknown`，排序时置于已知时间证据之后。

### M27.3 脱敏与完整性

- [x] 识别并处理绝对路径、串口设备名、工程名、备注和 provider 自由文本。
- [x] 支持 `local_export`（掩码）与 `provider_preview`（移除）两种 profile。
- [x] 记录 removed/masked 字段路径及原因，但不在审计记录中泄漏原值。
- [x] 计算 canonical JSON SHA-256；hash 输入排除 `integrity.sha256` 自身。
- [x] 在重新导入时同时验证 schema 和 SHA-256，拒绝被篡改的 snapshot。

### M27.4 Viewer 接入与 fixture

- [x] 提供从当前 session、选中时间窗和当前/指定 LA capture 创建 snapshot 的 API。
- [x] 提供 JSON 导入、canonical 导出和证据定位回调。
- [x] 建立正常、混合异常、wrap 和不完整 capture fixture。
- [x] 在 `ai_debug_validate.py` 中提供 `snapshot` 子命令。
- [x] 在 `justfile` 中提供组合验收入口 `m27-check`。

Viewer 接入 API：

```javascript

await api.createSession(options);
```

## 5. 测试矩阵

| 场景 | 实施结果 |
| --- | --- |
| 同输入重复构建 | 已验证完整 snapshot、canonical 内容和 hash 一致；默认 `created_at` 固定为 Unix epoch，也允许调用方显式传入 |
