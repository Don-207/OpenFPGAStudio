# M34 板级验证记录（2026-07-14）

- FPGA：Xilinx Kintex UltraScale `xcku5p_0`
- JTAG：Digilent FT232H，序列号 `210512180081`
- UART：CH340 `1a86:7523`
- Vivado：2024.2
- USER chain：USER2
- Mailbox：4096 bytes，build ID `0x4D340001`
- MPSSE TCK：约 6 MHz

## 构建与下载

- 完整综合、布局、布线和 bitstream 生成通过。
- 最终 route：0 failed/unrouted/partial nets。
- 最终镜像通过 JTAG 下载成功，startup status HIGH。
- JTAG Mailbox 映射为一个 `RAMB36E2`。

## 板级结果

- Header magic `OFJT`、transport v1、capabilities `0x000f` 校验通过。
- session ID `1`、buffer size `4096`、build ID `0x4D340001` 校验通过。
- 64-byte Payload 原始读取通过。
- commit 后 `read_count` 精确增加 64。
- 三轮 FT232H USB reset + TAP reset 重连通过：
  - `read_count 0 -> 64`
  - `read_count 64 -> 128`
  - `read_count 128 -> 192`
- 三轮 payload 前缀不同，无重复块。
- TAP reset 不再复位 Mailbox 数据面。
- Header 在扫描期间使用原子快照，未再出现计数不一致。
- 最终压力矩阵累计提交超过 14 KiB，`read_count` 到达 14,400；跨越
  8192-byte 环指针回卷点后，Header 和 commit 计数仍保持一致。
- 64/128/256/512/1024-byte 请求均完成真实 USER2 板级读取；其中持续有
  足够缓存数据时测得 64/128/256/512-byte 吞吐分别约为
  2.57/5.13/10.69/19.69 KB/s。
- 后续 1024/512-byte 档读取速度约 2.82 KB/s，此时已受板上约 2.5 KB/s
  数据生产速率限制，实际被拆成多个短事务，不能作为链路峰值。
- FTDI/MPSSE 批处理优化后，32-bit command DR 与首段 payload DR 在同一
  MPSSE/USB 批次连续执行；单批读回限制为 256 bytes，避免 IN FIFO 反压。
- 优化后实板以 256-byte 请求累计读取 4,159 bytes（17 个事务），耗时
  0.026739 s，测得 155,538.6 B/s（约 151.89 KiB/s），commit 计数一致。
  该短窗口链路峰值已越过 100 KB/s，但后续 512/1024-byte 档因缓存数据
  已耗尽而退化为短事务，不能据此评估对应整块吞吐。

## 验证期间修复

1. USER2 IR 必须在 DR scan 前显式选择。
2. TDO 不得由 SHIFT 组合门控，否则首位丢失。
3. BSCANE2 RESET 仅复位命令引擎，不得单独复位 Ring 读域。
4. Header 必须在响应扫描开始前冻结为一致快照。
5. Header 的 `write_count` 由 JTAG 域 `read_count + available_bytes` 构造，
   从结构上保证三个 ABI 字段属于同一快照。
6. commit 增量必须在加入 32-bit `read_count` 前显式零扩展；否则综合结果
   会在 13-bit 环指针的 8192 边界回卷。

- 最终镜像另做 10 秒回归：25,372 bytes、1,664 帧、约 2,537.1 B/s，
  checksum errors 0、version errors 0。
