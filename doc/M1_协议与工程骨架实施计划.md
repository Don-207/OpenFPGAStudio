# M1 协议与工程骨架实施计划

## 1. M1 目标

M1 的目标是把第一阶段从规划进入可开发状态，形成三条可并行推进的骨架：

- 协议骨架：固化 `OpenFPGA Debug Protocol v1`，让 FPGA 端、Qt Viewer、Web Viewer 使用同一份协议定义。
- RTL 骨架：建立 Debug Core 顶层端口、消息类型、基础模块边界。
- 上位机骨架：先提供 Web 版 Viewer 的最小可用界面，后续 Qt Viewer 可以复用相同协议逻辑。

M1 完成后，不要求 FPGA 端已经能输出真实 UART 波形，但必须让工程进入“可以按模块填实现”的状态。

## 3. M1 任务拆分

### M1.1 协议定义

任务：

- 固定帧头、版本、类型、长度、payload、checksum 字段。
- 明确所有多字节字段使用 little-endian。
- 固定第一批消息类型：Heartbeat、Debug Print、Event、Watch、Status。
- 明确 checksum 第一版使用 XOR，覆盖 `VER`、`TYPE`、`LEN` 和 `PAYLOAD`。
- 明确 payload 最大长度第一版限制为 32 字节。

验收：

- FPGA 和 Viewer 可以只看协议文档写出兼容的打包和解析逻辑。
- 文档包含至少 5 个消息类型的 payload 布局。

### M1.2 RTL 骨架

任务：

- 建立 `rtl/openfpga_debug` 目录。
- 定义消息类型和参数宽度。
- 定义 Debug Core 用户侧输入接口：
  - `event_valid/event_id/event_level/event_arg0`
  - `watch_valid/watch_id/watch_value`
  - `print_valid/print_id/print_arg0/print_arg1`
- 定义 UART 侧输出接口：
  - `uart_tx`
  - `busy`
  - `drop_count`
  - `packet_count`
- 建立 packetizer、timestamp、uart_tx 模块边界。

验收：

- 所有 RTL 文件可以被 Vivado 加入工程并通过语法检查。
- 顶层端口命名稳定，后续实现不需要推翻接口。

### M1.3 Web Viewer 骨架

任务：

- 建立 `tools/viewer/web` 静态页面。
- 使用 Web Serial API 提供串口连接和断开。
- 提供波特率选择：115200、921600、3000000。
- 实现协议 parser：
  - SOF 搜索
  - LEN 等待
  - checksum 验证
  - 错误计数
  - 未知类型统计
- 实现 Log、Event、Watch、Status 四个视图区。
- 提供“注入样例帧”按钮，用于无硬件时验证 UI。

验收：

- 直接打开 `tools/viewer/web/index.html` 后可以看到上位机界面。
- 点击样例帧后，日志、事件、Watch、Status 视图都有数据。
- 在支持 Web Serial 的浏览器中可以请求串口权限并读取字节流。

### M1.4 Qt Viewer 处理策略

M1 先不强制实现 Qt Viewer。原因是 Web Viewer 可以更快打通协议解析和界面验证，也便于后续调试板卡。

Qt Viewer 在 M2/M4 推进：

- M2：移植 Web parser 的状态机逻辑到 C++。
- M4：完善 Qt Model/View、导出、过滤和长时间运行性能。

## 4. 当前建议开发顺序

1. 完成协议文档。
2. 完成 Web Viewer parser 和样例帧注入。
3. 完成 RTL 顶层接口和空壳。
4. 用 Web Viewer 的样例帧反推 RTL packetizer 的输出格式。
5. 进入 M2：实现 packetizer 和 UART TX。

## 5. M1 完成定义

M1 完成需要同时满足：

- `OpenFPGA_Debug_Protocol_v1.md` 已存在。
- `rtl/openfpga_debug` 下已有顶层和核心模块边界。
- `sim/openfpga_debug` 下已有协议仿真入口。
- `tools/viewer/web/index.html` 可以打开，并能解析样例帧。
- 后续 M2 可以直接开始实现 UART TX、packetizer 和 parser 测试。
