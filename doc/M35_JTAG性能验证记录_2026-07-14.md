# M35 JTAG 性能验证记录（2026-07-14）

## 环境

- FPGA：Xilinx Kintex UltraScale，M34 build ID `0x4D340001`
- Cable：FT232H `0403:6014`
- USER chain：USER2
- Mailbox：4096 bytes，session 1
- Host：Direct FTDI/MPSSE backend

## 正确性

- Header magic、transport v1、capabilities、session 和 build ID 通过。
- 64-byte Payload 读取通过，前缀 `a501040ac1d27cfb0120c1d27cfb2ea5`。
- `read_count 24055 -> 24119`，commit 增量精确为 64。
- 常驻 Bridge 经原始 TCP 输出验证通过：3 秒收到 1 个 hello、321 个 DATA、6 个 status，
  客户端收到 7,615 payload bytes，无 reconnect、slow client 或 backend error。

## 性能原始结果

## 结论

- Direct FTDI backend、USER2 payload、commit 和 Bridge/socket 全链路均已真板闭环。
- 10 MHz、1024 B 的满 Mailbox 短窗口能力超过 100 KB/s，但只有 4 KiB，不能作为持续门禁。
- Bridge 状态在 3 秒窗口报告平均约 2,658 B/s，与当前业务数据源约 2.5 KB/s 一致；buffer
  很快降为 0。512/1024 B 的 6 MHz 数据也因供数不足退化为短块，不可用于块大小归因。
- 下一阻塞项是板端可控高速合法帧发生器。需要生成新的验证 bitstream，使 Mailbox 持续有
  至少 200 KB/s 数据，再执行 256/512/1024 B、可用 TCK、P50/P99、CPU 和 30 分钟长稳矩阵。

## 专用持续数据源复测

新增合法 HEARTBEAT 帧发生器，JTAG-only 性能模式每 50 个 100 MHz 时钟尝试提交一个字节，
Ring 满时保留当前字节且不提交重复 valid。专用 build ID 为 `0x4D350001`，UART 仍使用原业务
数据。仿真覆盖 checksum、连续帧和 backpressure。

- 综合：0 error、0 critical warning；一个 `BSCANE2`、一个 `RAMB36E2`。
- route：0 failed/unrouted/partial nets；最终估算 WNS `+3.638 ns`、WHS `+0.001 ns`。
- DRC/bitgen：通过；下载目标 `Digilent/210512180081`，startup HIGH。
- 下载后 Header：build ID `0x4D350001`、available 4095、overflow/drop 0。
- Payload 是连续合法 HEARTBEAT 帧，64-byte commit 正确。

仍属于 M36 发布门禁，尚未据本次短矩阵宣称完成。
