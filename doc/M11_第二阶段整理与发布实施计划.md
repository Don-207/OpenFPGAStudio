# M11 第二阶段整理与发布实施计划

## 1. 目标

M11 是第二阶段 Trace 的收口里程碑，目标不是继续扩展新功能，而是把 M7-M10 已完成的协议、Viewer、RTL Adapter、典型 probe 和板级 demo 整理成可复现、可交付、可继续迭代的发布包。

M11 需要回答三个问题：

- 用户如何把 Trace 接入自己的 FPGA 工程。
- 没有硬件、只有仿真、接上板卡三种情况下分别如何验收。
- 第二阶段 Trace 与第三阶段 Monitor 的边界在哪里。

## 2. 范围

必须完成：

- 补齐 `OpenFPGA_Trace_使用说明.md`，覆盖协议语义、RTL 接入、Viewer 使用、资源和限流建议。
- 补齐 `OpenFPGA_Studio_第二阶段Trace验证记录.md`，固化无硬件、仿真、Vivado elaboration、板级验证流程。
- 增加第二阶段 release checklist，明确发布前检查项。
- 更新 README，使 Trace 能力、M10/M11 验收命令和文档入口可见。
- 运行可在当前机器完成的回归测试，并记录结果。

暂不完成：

- PC 到 FPGA 的寄存器读写、在线变量修改和控制命令。这些归入 M12/M13 Monitor。
- DDR/PCIe/AXI 吞吐率、延迟分布等统计 profiler。
- 替代厂商 ILA 的连续波形采样。
- Trace 文本模板和复杂业务语义配置系统。

## 4. 验收矩阵

| 场景 | 命令或动作 | 期望 |
| --- | --- | --- |
| PC parser 回归 | `python tools\viewer\protocol_parser_test.py` | Trace begin/end/mark/value/drop 解析通过 |
| Web Viewer 无硬件验收 | 打开 `tools/viewer/web/index.html`，点击 `Inject Sample` | Trace 视图出现 Frame/DMA/FIFO/Interrupt 泳道和 timeout 高亮 |
| Web Viewer 性能冒烟 | `python tools\viewer\web\run_perf_test.py` | 大量 Trace 样例可渲染，checksum/sync/unknown 计数为 0 |
| M9 Adapter RTL | XSim 运行 `tb_openfpga_trace_adapter` | payload、握手、优先级和背压检查通过 |
| M10 Board demo RTL | XSim 运行 `tb_openfpga_debug_board_demo` | debug 帧和 trace 帧均能从 UART 流解析到 |
| Vivado elaboration | `vivado -mode batch -source prj/scripts/check_openfpga_trace_m10_elab.tcl` | RTL elaboration PASS |
| 板级长稳 | `tools\viewer\serial_validate.ps1` 或 Web Viewer 连接板卡 | checksum error 不持续增长，Trace 事件持续可见，drop_count 可解释 |

## 5. 发布步骤

1. 确认协议文档中 `0x10..0x14` Trace 类型与 RTL/View/parser 一致。
2. 运行 PC parser 回归。
3. 运行 Web Viewer 无硬件样例和性能冒烟。
4. 运行 M9/M10 RTL 仿真或 Vivado elaboration。
5. 使用当前 bitstream 或重新生成 bitstream 做板级 Trace 验收。
6. 更新验证记录，记录日期、环境、命令、结果和未关闭风险。
7. 检查 README 和使用说明中的命令是否仍能照抄执行。

## 6. 完成标准

- 文档能指导新用户完成 Trace 接入，不依赖口头上下文。
- 无硬件样例能展示完整 Trace 时间线。
- 仿真或 elaboration 能证明 RTL 结构和 payload 编码没有明显断裂。
- 板级记录说明 Trace 在真实 UART 链路上可观察。
- 明确第三阶段 Monitor 将从双向控制、寄存器访问和在线配置开始，不把这些内容塞进第二阶段。
