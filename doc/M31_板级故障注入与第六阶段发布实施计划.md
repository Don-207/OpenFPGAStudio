# M31 板级故障注入与第六阶段发布实施计划

M31 是第六阶段 AI Debug 的集成与发布里程碑。目标是在真实 board demo 或等价、可复现的硬件闭环中注入安全且可恢复的故障，对比本地规则、AI hypothesis 与人工确认根因，并完成隐私、降级、长稳和兼容发布门禁。

## 1. 目标与边界

- 至少验证 Transport、FIFO/backpressure、性能退化、LA trigger/完整性四类异常。
- 每个场景形成注入参数、原始证据、snapshot、规则结果、AI 结果、人工根因和恢复记录。
- 完成证据引用、误报/漏报、非法输出拦截和无 AI 降级评测。
- 完成前五阶段全量兼容回归、长稳、文档和发布 checklist。
- 证明关闭 AI 后不改变数据面、控制权限和板级性能。

M31 不为 AI 新增无保护写入口，不让模型直接操作硬件，不以真实 Provider 网络可用性作为本地 P0 发布阻塞项。

## 3. 前置条件

- M27 snapshot/schema/hash 回归通过。
- M28 P0 rules 和 golden cases 通过。
- M29 Mock Provider 的异常矩阵全部通过。
- M30 无硬件工作流和性能回归通过。
- 目标 bitstream、board、part、build ID、Viewer 版本和协议版本已记录。
- 故障注入只使用既有安全 demo 控制或专用测试模式，且有明确恢复步骤。

## 4. 板级故障场景

| 场景 | 注入方式 | 必需证据 | 预期诊断边界 |
| --- | --- | --- | --- |
| Transport error | 测试模式注入 malformed/checksum，或使用离线链路损坏 fixture | parser counters、transport health、时间窗 | 优先排查链路，不误判用户 RTL |
| FIFO backpressure | 降低 demo 消费速率或启用受控 stall | FIFO level、valid/ready、Profiler stall、Trace | 定位堵塞窗口，原因表述为候选 |
| Throughput/latency degradation | 调整 duty/stall 参数 | metric window、baseline、Trace duration、LA | 量化退化并关联相邻证据 |
| LA trigger missing | 配置不会命中的安全 trigger | arm/status、trigger config、信号活动 | 判断配置或未满足条件，不虚构触发 |
| LA data integrity | 离线丢弃一个采集 chunk 或测试读出故障模式 | chunk map、sample count、integrity | 要求重采，抑制强根因结论 |

若真实硬件不能安全注入 Transport/缺 chunk，可使用同一次板级采集派生的离线损坏副本，但验证记录必须明确标记“板级原始”与“离线派生”。

## 5. 单场景执行模板

1. 记录环境、版本、build ID、连接和正常基线。
2. 执行恢复点检查，确认故障可撤销。
3. 注入单一已知故障，记录参数和开始/结束 tick。
4. 采集 Debug/Trace/Monitor/Profiler/LA 与 transport counters。
5. 生成 snapshot，验证 schema、hash、完整性和脱敏。
6. 运行本地规则并保存 rule version 和 finding。
7. 使用 Mock Provider；可选使用真实 Provider 处理非敏感副本。
8. 由人工依据注入参数确认根因，记录命中、误报、漏报和信息缺口。
9. 按验证步骤恢复，确认指标和状态回到基线。
10. 导出 snapshot JSON、diagnosis JSON 和 Markdown 复盘报告。

## 6. 评测与发布门槛

| 指标 | P0 门槛 |
| --- | --- |
| Snapshot schema/hash | 所有发布样例 100% 通过 |
| 正式报告 evidence 引用 | 100% 有效 |
| P0 规则预期命中 | 100% |
| 正常基线 error 级误报 | 0 |
| Provider 非法输出拦截 | 自动向量 100% |
| 无 AI 降级 | 本地规则、证据跳转和导出可用 |

| 故障恢复 | 所有注入场景恢复到基线 |
| 前五阶段回归 | 全部通过 |

AI hypothesis 根因命中率作为评测记录，不作为唯一发布门槛；必须记录 provider/model/prompt 版本，禁止只保留“看起来正确”的样例。

## 7. 兼容、性能与长稳

- 运行 Debug、Trace、Monitor、Profiler、Logic Analyzer parser/RTL/Viewer 既有回归。
- AI Debug disabled 时确认无网络请求、无 Provider 初始化副作用。
- 确认 AI 层不能调用 Monitor write、LA command、program 或 build 接口。
- 大 evidence 包分析时持续接收数据，检查 drop、checksum、UI 卡顿和内存增长。
- 使用 Mock Provider 循环成功/超时/取消，检查请求和 history 资源释放。
- 建议完成不少于 30 分钟的持续接收与周期诊断；具体时长和数据量写入验证记录。

建议命令收口为：

```text
python tools/viewer/protocol_parser_test.py
python tools/viewer/ai_debug_validate.py all
python tools/viewer/web/run_perf_test.py
just ai-debug-regression
```

`just ai-debug-regression` 在实现阶段加入，并仅包含无硬件、无网络、可重复的发布门禁；板级和真实 Provider 命令应单独命名，避免误执行。

## 8. 文档与 Checklist

验证记录至少包含：

- 环境、日期、版本和命令。
- 每个场景的实际注入、人工根因、snapshot ID 和恢复结果。
- rule/provider/prompt/schema 版本。
- finding/hypothesis、有效证据、反证、误报、漏报和信息缺口。
- 离线、无密钥、timeout、非法输出、取消和迟到响应结果。
- 前五阶段回归及已知限制。

发布 checklist 至少检查：fixture、schema、golden、Mock、Viewer、隐私、凭据扫描、长稳、板级记录、README/使用说明和可复现命令。

## 9. 完成定义

M31 和第六阶段完成需同时满足：

- 至少四类异常形成可复现的闭环记录，且每类均能生成完整或明确标记不完整的 snapshot。
- 本地规则稳定命中预期事实，所有正式结论引用有效证据。
- AI 输出包含验证步骤；无依据或非法输出被降级/拒绝。
- 关闭 AI 后前五阶段功能、权限和性能无退化。
- 敏感字段和凭据泄漏为 0，外发必须显式授权。
- 自动回归、板级恢复、验证记录、发布 checklist、README 和使用说明齐备。

## 10. 后续方向

第六阶段发布后再评估构建日志、时序报告、厂商报告适配、项目知识库和半自动修复草案。任何自动执行能力仍需独立权限模型、预览、确认、回滚和审计设计，不纳入 M31 P0。
